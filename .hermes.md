# Hermes Rules for SocialListening

## Activation
These rules apply when working in `/home/tth-nguyenhung/agent/projects/SocialListening`.

## Canonical source of truth
These rules override older conflicting notes in chat transcripts and default assistant behavior. If changed, update `.hermes.md`, `AGENTS.md`, and `.agents/rules/pr-workflow.md` consistently.

## Operating mode
Short commands are first-class input.
- The user may describe a problem in one short sentence.
- Hermes must not require the user to write long prompts or full specs.
- Hermes must read project rules, inspect source, identify related files, make a short plan, and execute.

## Allowed autonomous actions (no confirmation required)
- Pull latest `main` and create a focused task branch.
- Read logs/status: Vercel preview (read-only), Render health/log/status (read-only), GitHub PR/checks.
- Inspect source and tests; identify files related to the issue.
- Fix small clear errors: TypeScript errors, import/name mismatch, missing types/fields, lint/build error, build/runtime error, API client calling wrong endpoint/name.
- Run relevant verification: `npm run type-check`, `npm run build`, `python -m compileall app`, relevant tests.
- Commit and push fixes.
- Update or add minimal docs when the task requires it.
- Create or update PR after branch has commits and basic verification; report PR URL.

## Required confirmation before acting
- Merge to `main`.
- Manual deploy/restart/migration on production.
- Changing env/secrets or production config.
- Any action with data loss risk.
- Adding large dependencies.
- Changing public API contracts or major architecture/product direction.
- Any step that cannot be justified from source/log inspection.

## Autonomous Fix Loop behavior
When the user gives a coding/repair/task instruction:
1. Read project rules first.
2. Inspect source and related files.
3. Make the smallest possible plan.
4. Implement, verify, commit, push.
5. If a check fails, inspect log/file/line, fix, commit, push, recheck.
6. Repeat until checks pass or a hard blocker is encountered.
7. Report clearly.

For known short commands:
- `tự fix đi` — start/continue autonomous fix loop on current branch/task/PR
- `làm tiếp đi` — continue current unfinished task safely
- `check deploy đi` — read-only platform status/log/health check
- `tạo PR đi` — create/update PR into main; do not merge
- `merge đi` — merge only if checks pass or user accepts risk; prefer squash merge
- `dừng lại` — stop current task; report status without destructive actions

## Non-goals and constraints
- No fake data, fake APIs, fake responses, dummy UI, fake AI outputs.
- No broad refactors unrelated to the task.
- No changes to unrelated modules, pages, APIs, models, migrations, UI flows, RBAC, dashboard, scanner, reports, login/auth, settings unless directly required.
- Do not commit `.env`, secrets, API keys, local DB files, caches, logs, temp scripts, unrelated docs, `deploy_trigger.txt`, `frontend/tsconfig.tsbuildinfo`.

## Branch and PR workflow
- Always create a feature/fix branch from latest `main`.
- Never push directly to `main`.
- Commit only intended files.
- Before commit: run `git status --short`, review `git diff --stat`, inspect changed files.
- Create PR into `main`.
- Do not merge automatically.
- Default after coding: stop and ask for confirmation before PR/merge unless user explicitly requests otherwise.

## Completion report (required)
For every task, keep the full required reporting fields, but the **default Telegram/report style** must follow the **Concise Reporting Style** section below unless the user asks for detailed logs.

- Branch
- Commit hashes
- PR URL if created
- Files changed
- Docs updated
- Tests/checks run
- Vercel/Render status
- Risks/notes
- Ask the user only when merge/deploy or a risky decision is needed

## Concise Reporting Style
Default output style for Telegram and for reports/questions back to the user.

Rules:
- Ngắn gọn, tổng quan, đi thẳng vấn đề.
- Ưu tiên format dễ đọc.
- Có thể dùng emoji nhẹ: ✅ ⚠️ ❌ 🔧 🚀, không lạm dụng.

For success:
- ✅ Xong
- Branch / commit / PR nếu có
- Tests/checks
- Còn rủi ro gì không

For asking:
- Chỉ hỏi đúng 1 câu quan trọng nhất.

Rules:
- Chỉ gửi log/chi tiết dài khi user yêu cầu hoặc khi lỗi thực sự cần debug.
- Nếu có lỗi, chỉ trích dòng lỗi chính + file/dòng liên quan.
- Không nói dài kiểu “mình sẽ tiếp tục…” nếu không cần.
- Với lỗi nhỏ Hermes tự xử lý được: không hỏi, chỉ báo kết quả cuối vòng fix.
- Với quyết định lớn (merge/deploy/restart/migration/env/secrets): hỏi ngắn gọn.

## Backend/frontend verification preferences
- Frontend: `npm run type-check`, `npm run build`, or existing frontend check command.
- Backend: `python -m compileall app`, existing relevant tests.
- Full-stack: frontend + backend checks when API contract changes.

## Production behavior
- Do not run `deploy.bat`, production migrations, or production restarts unless explicitly requested.
- Render/Vercel may auto-deploy after merge to `main` by default.
