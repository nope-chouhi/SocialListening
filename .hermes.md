# .hermes.md - SocialListening

## Branch workflow
- Always use a dedicated branch from latest `main`.
- Commit and push that branch before any merge-related step.

## Stop-and-confirm rule
After pushing a branch, Hermes must stop and ask for confirmation. Do not create a PR until the user says yes.

## PR rule
Do not merge any PR unless the user explicitly confirms and checks have passed.

## Change scope
- Preserve stable completed modules.
- Do not refactor unrelated modules.
- Fix only the requested issue.

## No fake behavior
Do not create fake UI, API responses, mentions, charts, scan results, reports, or exports.

## Environment and secrets
- Do not hardcode local URLs, secrets, tokens, or environment-specific values.
- Do not commit `.env`, secrets, API keys, local database files, logs, or temporary artifacts.

## Documentation rule
Every code task must add or update at least one relevant `.md` file when changes affect behavior, API, workflow, or setup. No fake docs.

## Deployment
Do not run `deploy.bat`, deploy scripts, production migrations, or production restarts unless explicitly requested.
