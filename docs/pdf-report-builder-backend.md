# PDF Report Builder Backend Integration

## Overview
This document describes the implementation details of integrating the frontend PDF Report Builder UI with the real backend. The goal was to eliminate any "frontend-only mock behavior" when exporting PDFs, ensuring that the PDF generated by the backend perfectly reflects the builder configurations chosen by the user in the UI.

## Database & Models
- **ReportExport Model**: Added `builder_config` (JSON) to the `ReportExport` table in `backend/app/models/report.py`. This column stores all options (colors, fonts, theme, section ordering, and toggles).
- **Alembic Migration**: Generated and applied migration `bdd2e5fc2cee_add_builder_config_to_report_exports.py` to add the new column to SQLite without affecting existing tables.

## Schemas
- **ReportBuilderConfig**: Added to `backend/app/schemas/report.py` to validate incoming UI state when requesting a new export.

## Logo Upload & Customization
- **`POST /api/reports/pdf/logo`**: Accepts an `UploadFile` (JPEG/PNG) and stores it locally to `data/uploads/logos/`.
  - Max file size: 5MB.
  - Returns a `logo_path` which is then mapped into `ReportBuilderConfig`.
  - **Storage Limitation**: On Render, local file storage is ephemeral. Thus, uploaded logos will not persist across container restarts. Production implementations should eventually mount persistent disks or switch to S3.
- **Preview Consistency**: The React frontend preview was rewritten to natively map over the dynamically sorted `sections` array instead of hardcoded evaluations. This ensures what the user sees in the browser matches the generated PDF block sequence precisely.

## APIs
- **`POST /api/reports/export/{report_type}`**: Updated to accept `builder_config` in the request body, saving it to `export_job.builder_config`.
- **Preview**: Discovered that the frontend's preview is natively rendered via React using `summary-data` and component local state, rather than requesting a rendered artifact from the backend. Thus, no new `/api/reports/preview` was strictly necessary; we simply map the same React state parameters into the actual `builder_config` payload on export.

## Background Generation Logic
- **`ExportService.process_export`**: Updated to read `export_job.builder_config`.
  - Date boundaries (`date_from`, `date_to`) are parsed and applied explicitly to filter all data subsets (mentions, alerts, incidents).
  - Passes the `builder_config` into `pdf_data` for downstream rendering.
- **`PDFGenerator.generate_project_summary`**: Updated to utilize `builder_config`.
  - Parses and maps custom font styles (`font-sans`, `font-serif`, `font-mono`) to standard PDF font families (Helvetica, Times, Courier).
  - Dynamically builds the PDF sections based on `builder_config.sections` ordering and `enabled` status.
  - Applies custom background themes and accent colors cleanly mapping to standard ReportLab APIs.

## Frontend Updates
- **`api.ts`**: Updated `requestExport` to accept and serialize the `builder_config` JSON payload.
- **`ReportsPage` (`page.tsx`)**:
  - Bound `handleExport` to send actual React state configurations to the API.
  - Added native inline section-reordering controls (Up/Down) using `lucide-react`'s `ChevronUp` and `ChevronDown`, enabling section drag/drop alternative without bloating bundle size.
